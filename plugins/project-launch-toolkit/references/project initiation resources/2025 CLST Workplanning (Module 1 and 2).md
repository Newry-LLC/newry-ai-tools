# Module 1: Workplanning

## Every project starts with a problem statement, which drives the issue tree

Problem statement template includes: Client Business Context, Stakeholders (Sponsor + Direct Contact), Scope (In/Out), Project Objective (SMART), Risks, Key Milestones.

The issue tree then flows directly into the workplan via the issue → hypothesis → analysis → activities chain.

---

## From Issue Tree to Workplan

**Defining the work** — each issue or hypothesis should articulate:
- Desired end product
- Analysis required to produce the end product
- Sources to include
- Timing and responsibility

**Sequencing the work** — workplan should:
- Match available resources against key engagement milestones (e.g., progress reviews)
- Align with the overall pacing of the engagement (weekly or twice-weekly meetings)

---

## Elements of a Successful Workplan

**Staffing / Roles / Assignments**
- Specifies team member roles and responsibilities day-to-day
- When possible, makes assignments to meet personal learning objectives

**Time Management**
- Maps all major activities to the issue analysis and project timeline
- Allows for time to interact with client/ED/editors and implement feedback
- Enables EM to keep team members on track

**Financial Budget Management**
- Travel and typical project expenses
- Secondary research / purchased reports
- SMEs, contractors, or other planned services
- Trips to the client

---

## Ways to Delegate

| Organizing Principle | Example |
|---|---|
| Activity | Bob does interviews, Sally does model and forecast |
| Market Segment | Bob does automotive, Sally does aerospace |
| Key Question | Bob determines value prop; Sally determines lead adopters |

**Elements to keep in mind:**
- Get your team to speak up about their learning goals
- Articulate expectations on analysis, end products, and deadlines as clearly as possible
- Plan internal meetings to touch base or report out
- Get a sense for other time commitments (vacation, training)

---

## Three Time Horizons for Workplanning

1. What is expected at the end of the engagement (high-level, full project length; revised only for major events)
2. What is expected at key progress reviews (milestone-level)
3. What is due daily and/or weekly at team meetings (granular, covers 2–3 weeks; revised constantly as insights come in)

Workplan formats: ThinkCell Gantt, Excel Gantt. Good workplan examples on SharePoint: Carrier (modeling), CGS (growth engine), Halocarbon (segmentation & strategy), CPO (opportunity analysis).

---

## Shorter Projects

Projects that may not require in-depth workplans:
- Roadmapping, benchmarking (limited set of activities)
- M&A Screening, SONAR, Surveys (same approach every time)

---

## What Happens When You Don't Make a Workplan

- You duplicate each other's work or do analysis that doesn't get used
- You don't source or conduct interviews effectively
- You end up staying up late the night before an update
- You don't hit budget targets (affects EBITDA, affects bonus)
- Team members don't get to work on development goals (comes up in ACC)
- You forget to answer a key question
- Not enough time for thorough synthesis → lower-quality client output
- The client gets mad

---

## Weekly Synthesis

Teams should synthesize findings and update issue tree hypotheses weekly, articulating the "Day 1" or "Week 1" answer. Approaches: weekly memos, Slack call-in posts.

# Module 2: Execution & Organization

## At the Beginning of a Project

**Project Launch documents:**
- Project Initiation template: problem statement, issue tree, work plan, team goals, firm & client goals, team expectations
- Sample Project Launch: example workplans from past projects

**Airtable:** Enter fees and expenses; SharePoint auto-updates with project code and folders.

**Week 1 Scheduling checklist:**
- Regular client updates (1–2 weeks or as needed)
- Major client updates (aim for in-person)
- Peer review(s) for large/new/ambiguous projects
- Recurring internal team meetings (may include editors)
- Recurring 1:1s (ED:EM, EM:Team)
- Discussions with internal SMEs

---

## Ghost Deck (first 1–3 weeks)

- SOF outline that matches deck flow
- Outline of all slides with hypothesized headers and content ideas
- A formatted money slide
- Review with EDs and editors

---

## Standard Weekly Team Meeting

**Objectives:** Align on project objectives / issue tree / key questions; align on progress; identify data/analysis gaps; troubleshoot issues.

**Agenda:**
1. SoF — review and revise current version (EM and team members draft and compare)
2. Progress update — interviews, model progress, etc.
3. Issues — each team member shares current/anticipated issues; prioritize and problem-solve
4. Workplan — review and set next steps for the week

---

## Internal Resources

| Resource | Approach |
|---|---|
| ED/EM/LC | Establish roles and expectations at project start |
| Peer Reviews | Use early to work through complex issues and frame problem-solving; common trip wire: waiting too late |
| Editorial Support | Establish contact early; give advance notice for documents; put time on calendar for review |
| Contractors | Use when capacity constrained, internal resources unavailable, ED approves |
| SME Contractors | Connect team to industry contacts; can review/amend synthesized work; discuss with ED |

---

## Execution Challenges and Solutions

| Challenge | Solution |
|---|---|
| Client changes timeline | Weekly synthesis for up-to-date answer at any time |
| Client team changes | Kickoff with new stakeholders mid-project |
| Client will not provide needed data | Communicate impact to timeline/budget; use phase 0 |
| Multiple stakeholders not aligned | Schedule calls with dissenters; informal calls smooth relations |
| Challenges obtaining data | Find creative solution; plan a backup at project start |
| Underperforming consultant | Plan additional check-ins; revise delegation |
| Staffing changes | Add resources; reprioritize workstreams |
| Scope changes | Internal meeting to review revised workplan |
| Workplan not revisited | Set agenda item in weekly team meeting |

---

## Project Organization Best Practices

**Documentation:** All docs saved on SharePoint with naming convention; ghost deck in first third; research deck; notes in text boxes (unique color per team member); well-labeled, sourced, color-coded spreadsheets.

**Communication:** Overschedule touchpoints and cancel as needed; agenda for all meetings; project Slack channel; keep 1:1 in DMs.

**Workplanning:** Calendar-driven; work backwards from final; keep calendars current; plan editor/ED/peer reviewer time far in advance; keep a running to-do list organized by priority; block focus time.

## McKinsey's "Chunky Workplan, Lean Project Plan"

- **Chunky workplan:** Short and highly specific; covers 2–3 weeks; revised constantly; assigns responsibility for specific tasks; sets short-term deadlines
- **Lean project plan:** High level; covers the full project including fixed milestone dates; revised only when major events rescheduled or disruptive new information
